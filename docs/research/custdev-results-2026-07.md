# Результаты кастдева №1 (4 респондента) — выводы для продукта

Дата: 2026-07-06. Источник: ответы на [короткий бриф](custdev-brief-2026-07.md) (10 вопросов).
Выборка: 4 человека из ближнего круга — не статистика, но паттерны консистентны между собой и с рыночным анализом.

## Респонденты

- **A — автор/power user**: 4 скила в проде 4–5 месяцев + большой плагин в тесте; переписывал MCP под графовую БД; держит сетап «лаконичным»; сканирует чужое на малварь руками; чужое не ставит — «слизывает паттерны» в своё.
- **B — активный пользователь**: скилы/воркфлоу/MCP; проверяет чужие скилы, прося агента (Codex) найти опасное; смотрит наличие автообновления; «нет нормального поиска по скиллам»; готова платить за маркетплейс с доверием/версиями/отзывами/понятным install-update; исходники читает всегда; «я жадная» (не публикует бесплатно).
- **C — новичок**: сам не пишет; ставит из TG-каналов и по рекомендации («Никита дал скилл — усиленный superpowers»); доверяет «если лежит на GitHub»; к оплате относится легко — «копейки по сравнению с токенами».
- **D — casual**: скилы и немного субагентов; главная боль — потеря контекста/рестарты; ищет через Google/GitHub, часто агент сам находит по ссылке; ставит «рекомендованное» без проверок; хочет «шильдик approved» и соответствие стандарту.

## Ключевые паттерны

1. **Никто не платил (0/4), но готовность есть у двоих.** B: заплатила бы за маркетплейс с доверием, версиями, отзывами и понятным install/update. C: платить легко, если «копейки». → Входные цены $5–19, не $50; выручка первого года не с индивидуальных продаж.
2. **Проверка безопасности — театр на всех уровнях.** A сканирует руками (единственный), B и D делегируют проверку агенту, C не проверяет вообще и знает, что это неправильно. Продукт должен делать проверку ЗА них и показывать вердикт.
3. **«Бенчмарк ещё изобрести нужно» (A, дословно).** Для установки без чтения исходников A хочет: гарант безопасности + бенчмарк в сравнении с аналогами. Это прямой запрос на категорийные eval-сьюты и leaderboard «этот скил vs 3 аналога».
4. **Исходники священны для продвинутых.** B: «без вариантов — всегда проверяю исходники». → Платное НЕ значит закрытое: продаём обновления/гарантию/исполнение, исходники остаются читаемыми. Закрытый код = потеря всего верхнего сегмента.
5. **Гигиена сетапа — невидимая боль №1 продвинутых.** A: раздувание количества скилов → деградация триггеров + сожранный контекст; решение — держать сетап лаконичным, ставить под задачу и удалять. D: потеря контекста/рестарты — самая частая проблема. Никто на рынке это не решает.
6. **Дистрибуция = рекомендации + TG/Threads.** Ставят то, что дал знакомый или TG-канал (C: «Никита дал скилл»; D: «рекомендованный»). GitHub — место хранения, не место выбора. → Creator-led GTM подтверждён этой выборкой.
7. **Установка не работает «из коробки».** B: Codex не подхватил пути, 15–20 мин чинить; «часто перебор методом тыка, пока не заработает». D: инструкции разные, «ладно если вообще есть». → Нужен verified-install бейдж per client и пост-инсталл проверка.
8. **Поиск болит.** B: «бесит, что нет нормального поиска по скиллам». Наш registry search — прямое попадание, но должен быть лучше GitHub-поиска ощутимо.
9. **Авторы не публикуют из-за запутанности, а не жадности** (кроме B — та из жадности). A: скилы «слишком разрозненные, встраивались в воркфлоу суперпауерса» — его наработки имеют зависимости от чужого плагина, распутывать лень. → Нужен tooling «извлечь и упаковать» + поле зависимостей в манифесте. B-жадность лечится деньгами — кандидат №1 в платные авторы.
10. **Teams-боль в выборке не подтверждена** — команды маленькие, шарят «вручную», «в чатике», «на митах». Гейт M2 работает правильно: сигнала нет, значит нужны респонденты из команд 5+ человек. Отдельно: изучить компании покрупнее следующей волной кастдева.
11. **Незакрытый спрос — конкретный**: B ищет «идеальный скилл на создание спек — так и не нашла, хотя он точно должен быть». Готовый кандидат в первый bounty и seed-контент.
12. **Агент — уже установщик.** D: «если агент сам найти не может — гуглишь, отдаёшь ссылку на гитхаб». Агент-first поверхность (llms.txt, MCP, hh) подтверждена: агент должен находить и проверять харнесы у нас, а не гуглить.

## Что поменять в концепте/плане

1. **Зафиксировать: paid = open source.** В M1 запретить скрытые исходники для платных харнесов; монетизация — обновления/gate/endpoint (моделям M2/M3/M1 это и так соответствует, но теперь это правило продукта, а не опция). — из B#10.
2. **Ценовой якорь вниз (гипотеза, n=1)**: one-time $5–19, подписки $3–9/мес. Основание слабое — один casual-респондент («копейки») и нулевая база платящих; в M1 проверяем ценовыми экспериментами на реальных покупках, в прайс жёстко не зашиваем. — из C#9.
3. **Teams: вывод «спроса нет» отозван** (поправка фаундера + ошибка выборки). В выборке не было ни одной команды 5+ — она физически не могла подтвердить или опровергнуть Teams-боль; отсутствие сигнала ≠ негативный сигнал. **Teams остаётся в плане по расписанию (M2)**; рыночное обоснование B2B из конкурентного анализа действует независимо. Кастдев команд 5+ теперь калибрует фичи M2, а не решение «делать или нет».
4. **GTM: X/Reddit остаются основными каналами (глобальный рынок), TG добавляется для RU-сегмента.** Вся выборка русскоязычная — TG-уклон это артефакт выборки, не рынка. Что подтверждено надёжно — сама механика рекомендательных цепочек: ставят то, что дал знакомый. Анкор-кандидат №1 — автор «усиленного superpowers» (Никита из ответов C, де-факто уже дистрибьютор). B — кандидат в первые платные авторы («я жадная» + неопубликованные наработки). — из C#3, A#5.
5. **Первый bounty засеять сразу**: «скилл на создание спек» — подтверждённый неудовлетворённый спрос. Не ждать M4: можно провести вручную (заказчик B, исполнитель A?) как эксперимент до всякой эскроу-инфраструктуры. — из B#8.
6. **Trust-панель пересобрать вокруг трёх вопросов респондентов**: «безопасно?» (Н1-скан + permissions), «работает?» (verified-install Н7 + evals), «лучше ли аналогов?» (Н2-бенчмарк). Порядок именно такой — это дословно их критерии установки.

## Что подтвердилось (менять не надо)

- Agent-first поверхность (агент уже ищет и ставит сам) — D#3.
- Install-first флоу и Install Center per client — B#2/#5, D#5.
- Подписка на обновления как ценность: B прямо смотрит «есть ли автообновление» как плюс — M2-модель.
- Creator-led GTM через доверенных людей с аудиторией — C#3, D#4.
- Поиск как core-фича реестра — B#5.
- Секвенирование «доверие до денег» — 0/4 платили, готовность появляется только после доверия (B#9: «если там есть доверие, версии, отзывы»).

## Критическая переоценка достоверности (n=4 — честность выборки)

| Вывод | Базис | Уверенность | Действие |
|---|---|---|---|
| Security-scan нужен продукту | 4/4 консистентно + рыночная логика | Высокая | Делаем в M0 |
| Discovery/поиск болит | 2/4 явно + рыночный анализ | Высокая | Уже в плане |
| Install-friction (не работает из коробки) | 2/4 + рыночная логика | Высокая | Verified-install в M1 |
| Рекомендательные цепочки как канал | 3/4 | Высокая (механика), канал TG — артефакт RU-выборки | X/Reddit основные, TG для RU |
| Paid = open source | 1/4 (но целевой power-сегмент) | Средняя, дёшево соблюсти | Правило продукта |
| Бенчмарк-сравнение | 1/4 (автор) | Средняя | M1+, начать с 1–2 категорий |
| Context-cost / гигиена сетапа | 1/4 (+смежная боль D) | Гипотеза | Дёшево — ставим и меряем реакцию |
| Временная установка | 1/4 | Гипотеза | Продвигается как инфраструктура плагина, не как отдельная фича |

## Плагин onlyharness (решение фаундера, подтверждаемое ответами)

Плагин, который при установке сам подбирает под текущую задачу скилл/воркфлоу из реестра и применяет его (бесплатные — сразу, платные — через оплату). Ответы прямо поддерживают паттерн: D — «если агент сам найти не может — гуглишь, отдаёшь ссылку» (агент уже установщик, дадим ему каталог); B — просит агента проверить скилл на опасное (плагин отдаёт security-verdict из реестра). Добавлен в мастер-план как Milestone MP; безопасность: только просканированное, только с подтверждением пользователя, permissions всегда на экране.
