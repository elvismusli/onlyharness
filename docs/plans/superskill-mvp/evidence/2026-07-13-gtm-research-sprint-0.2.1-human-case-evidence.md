## Case 1 — normal design-partner sprint

Case ID: `human-normal-gtm-sprint-001`

### Public-safe input

> Design a two-week GTM research sprint for a reviewed agent-capability registry. Known
> facts: 12 capabilities are selected but unreviewed, no release is approved, the product
> supports Claude Code and Codex, and the pilot goal is five design partners completing
> one repeated task twice. Use only these facts. Define the target segment, account
> criteria, pain hypotheses, testable outreach angles and proof plan. Do not identify or
> contact real people.

### AI-generated candidate output

#### Target segment

Start with small product or engineering teams that already use both Claude Code and
Codex, repeat at least one review or research workflow weekly, and can assign a human to
judge the same output twice during the pilot.

Exclude teams that need an approved production catalog now, require autonomous external
actions, or cannot provide a public-safe test task. The current zero-approved state makes
those needs incompatible with this candidate-only pilot.

#### Account qualification criteria

An account candidate is eligible only when public-safe or voluntarily supplied evidence
supports all of these criteria:

1. both named clients are in active use;
2. one repeated weekly task can be described without private customer data;
3. one accountable reviewer can score two attempts;
4. the team accepts candidate-only language and no managed production activation;
5. the team can report success, failure and limitations without sharing prompts or
   credentials.

No account names are proposed because the input contains no source-backed account facts.

#### Pain hypotheses and testable angles

| Hypothesis | Public-safe angle to test | Disconfirming signal |
|---|---|---|
| Teams duplicate workflow instructions across clients. | “Test one reviewed instruction set in Claude Code and Codex without rewriting the task.” | The team already has a stable shared workflow and sees no duplication cost. |
| Teams cannot distinguish selected candidates from approved capabilities. | “Evaluate an exact candidate release with explicit review state and limitations.” | The team does not need provenance or approval-state clarity. |
| Teams lack a repeatable evidence loop for capability quality. | “Run the same bounded task twice and record human verdicts and limitations.” | The team cannot name a repeated task or accountable reviewer. |

These are hypotheses, not confirmed pains. Any future outreach copy must preserve the
facts that approval is zero and managed activation is unavailable.

#### Two-week proof plan

- Days 1–2: collect no more than 15 account candidates using the criteria above and
  record a public-safe source or voluntary qualification note for every criterion.
- Days 3–4: ask an authorized operator to choose five design partners; this resource
  does not contact them.
- Days 5–10: each partner runs one bounded task twice in both clients and records whether
  the workflow completed, whether behavior matched permissions, and whether it helped.
- Days 11–12: reconcile failures and limitation codes without converting candidate
  results into approval claims.
- Days 13–14: report completion count, repeated-use count, client-specific failures and
  unresolved blockers. The sprint passes its stated proof target only if five partners
  complete both attempts; it does not by itself approve a capability.
