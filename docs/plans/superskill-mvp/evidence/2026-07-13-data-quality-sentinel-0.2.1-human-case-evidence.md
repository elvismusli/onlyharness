## Case 1 — normal launch-metric audit

Case ID: `human-normal-data-quality-001`

### Public-safe input

> Decide whether an 18% week-over-week ARR increase supports launching a new plan. The
> dashboard compares Monday–Sunday weeks and was refreshed Monday at 06:00 UTC. Billing
> exports arrive two days late. Last week the trial-conversion denominator changed from
> all started trials to trials that reached onboarding step two. The dashboard owner has
> not reconciled the new cut with billing. No row-level data is included.

### AI-generated candidate output

#### Verdict

**NO-GO on using the reported 18% increase to authorize the launch.** The business may
evaluate other launch evidence, but this metric is blocked by freshness and denominator
comparability defects.

#### Source and metric map

| Metric or evidence | Source / owner | Window and filters | Quality state |
|---|---|---|---|
| Dashboard ARR | Modeled dashboard; owner mentioned but not identified | Monday–Sunday; refresh Monday 06:00 UTC | Billing source can be incomplete by two days; calculation and currency treatment are not supplied. |
| Billing ARR | Billing export | Export lag: two days | Authoritative relationship is not explicitly supplied; current-week completeness is unknown. |
| Trial conversion | Dashboard calculation | Prior denominator: all started trials; current denominator: trials reaching onboarding step two | Not comparable across the stated weeks. |
| Launch decision | Accountable owner not supplied | Threshold beyond “18% increase” not supplied | Decision rule is incomplete. |

#### Blocking findings

1. The Monday dashboard can omit Saturday and Sunday billing movements because the export
   lags by two days.
2. Trial conversion changed population, so any week-over-week conversion movement cannot
   be interpreted as behavioral change without a backfilled common denominator.
3. ARR and billing have not been reconciled, so the 18% number lacks a verified source
   bridge.
4. No launch threshold or accountable metric owner is provided.

#### Smallest useful validation

An authorized analyst should run a read-only reconciliation after the lag window closes:

- recompute both weeks’ ARR from the same finalized billing snapshot, currency rule,
  cancellation rule and account population;
- recompute both weeks’ trial conversion twice using one common denominator definition;
- report record count, total ARR and unmatched-account count for dashboard versus billing;
- require the metric owner to explain every material difference before reissuing the
  launch recommendation.

This output did not run a query or inspect a dashboard. It does not claim that the launch
itself is unsafe; it says the supplied 18% metric is not decision-grade evidence yet.
