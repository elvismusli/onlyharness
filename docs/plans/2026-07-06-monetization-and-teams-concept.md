# Концепт: монетизация OnlyHarness + командные воркспейсы

Дата: 2026-07-06. Статус: концепт для обсуждения с партнёрами (не implementation-план).
Основа: [monetization-competitors-2026-07.md](../research/monetization-competitors-2026-07.md) (рынок), [custdev-brief-2026-07.md](../research/custdev-brief-2026-07.md) (вопросы для проверки гипотез), [agent-first-surface](2026-07-06-agent-first-surface.md) и [install-first-showroom](2026-07-06-install-first-showroom-flow.md) (продуктовая рамка), техническая инвентаризация кода от 2026-07-06 (внизу раздела 3).

## 0. Позиция одним абзацем

OnlyHarness продаёт **не файлы, а четыре вещи, которые нельзя скопировать вместе с файлом**: гарантию качества (evals/gate), поток обновлений и совместимости, исполнение (hosted endpoints), приватную дистрибуцию для команд. Покупатель — и человек, и его агент (llms.txt + HTTP 402). Комиссия двухуровневая по модели Whop: ~5% на продажах, куда трафик привёл сам креатор, 15% на продажах через наш каталог — против плоских 20–30% у Agensi/MCPize/Apify. GTM — через креаторов с аудиторией на X/Reddit (+TG для RU-сегмента), которым мы закрываем витрину и весь финтех до сетлмента (раздел 5): они продвигают свои системы, зарабатывают и приводят комьюнити. Отдельная ключевая поверхность — **плагин onlyharness** (раздел 6): агент пользователя сам подбирает харнес под задачу из реестра и применяет его, с покупкой для платных.

## 1. Итоговая модель монетизации

### Сводка: модели, момент оплаты, канал

| # | Модель | Что покупают | Момент оплаты | Рельсы | Фаза |
|---|---|---|---|---|---|
| M2 | Подписка на обновления | Канал версий + совместимость | Pull новой версии / месяц | Paddle (фиат) + x402 | 1 |
| M2b | One-time покупка | Разовый доступ к платному харнесу | При pull (402 → оплата → архив) | Paddle + x402 | 1 |
| M3 | **Gate-эскроу** («плати, если работает») | Гарантированный результат | Резерв при pull, списание при прохождении `hh gate` | эскроу поверх M2b | 3 |
| M1 | Hosted endpoint (per-call) | Исполнение «мозга» автора | В момент вызова (x402 USDC) | x402 | 4 |
| M4 | Bounty | Заказная разработка харнеса | Эскроу при старте, релиз по приёмочным evals | эскроу | 4 |
| M5 | Team workspace | Приватная дистрибуция для команды | Подписка $/seat/мес | Paddle (инвойсы) | 2 |

### Тарифная сетка (черновик)

- **Free** — публичные харнесы, безлимитный pull, локальные evals, публикация. Навсегда: это маховик доверия.
- **Creator** — бесплатно; платные харнесы включаются полем `pricing` в манифесте. Автор получает **95% на продажах по своей ссылке** (attribution по referral) и **85% через каталог**; анкор-креаторам первого полугода — 100% по своей ссылке (мы зарабатываем на платёжной марже, т.к. сетлмент наш). Выплаты: USDC на кошелёк (сразу) или фиат через нашу фин-инфру.
- **Team** — ~$12–15/seat/мес, первые 3 места бесплатно (для маховика). Приватный воркспейс, setup-бандлы, аудит.
- **Enterprise** — SSO/SCIM, self-hosted registry (`HH_REGISTRY_URL` уже позволяет — продаём поддержку и лицензию), custom.

### Двухканальные платёжные рельсы

Один ответ `402 Payment Required` обслуживает обоих покупателей:

```
GET /repos/acme/deploy-checklist/archive
→ 402 {
    "x402": { scheme: "exact", network: "base", asset: "USDC", amount: "4.00", payTo: ... },
    "checkout_url": "https://onlyharness.com/buy/acme/deploy-checklist"   ← для человека
  }
```

Агент платит USDC через x402 facilitator (CDP), человек уходит в Paddle/LemonSqueezy checkout (MoR закрывает VAT/инвойсы). Оба пути кончаются одинаково: записью в `entitlements`.

Почему оба канала обязательны: x402 даёт agent-native покупку и не требует Stripe-энтити, но реальный спрос на микроплатежи пока тонкий (падение объёмов на 92% с пика, CoinDesk); корпораты платят только фиатом с инвойсом. Ставим на оба, основную выручку первого года ждём с фиата (M5 Teams).

## 2. Командные воркспейсы (B2B концепт)

### Философия

По install-first концепции OnlyHarness — не workspace, где работают, а showroom + trust + install. Командный воркспейс это НЕ меняет: он — **приватный слой дистрибуции и доверия**. Команда продолжает работать в своих Claude Code/Cursor; воркспейс — их внутренний магазин харнесов с контролем качества.

### JTBD команды (проверяем вопросами 7 и 22–26 кастдева)

1. «Новый человек получает весь наш агентский сетап одной командой» — сегодня это полдня ручного онбординга.
2. «Я починил харнес — вся команда получила обновление и узнала об этом» — сегодня: git pull, если повезёт.
3. «Я вижу, что наши харнесы могут делать (shell, network, money_movement) и кто что тянет» — сегодня: никто не видит.
4. «Взять публичный харнес внутрь безопасно, с provenance» — сегодня: copy-paste без истории.

### Механика

- **Org = namespace**: `@acme/deploy-checklist`. Роли: owner / admin / publisher / member. Приватные харнесы `visibility: org` — архив отдаётся только членам.
- **Win98-метафора: Network Neighborhood.** Иконка «сеть» на рабочем столе открывает воркспейс компании — тот же весёлый каталог, но с замочками на карточках и своей темой окна. Внутренний каталог, который выглядит как публичная витрина, — сам по себе продающая фича (ни у одного private registry нет вменяемой витрины).
- **Setup-бандл**: `hh setup @acme` — ставит стандартный набор команды одной командой (бандл-манифест: список харнесов + пины версий + конфиги под Claude Code/Codex/Cursor). Это ответ на JTBD-1 и главный wow-момент демо.
- **Update flow**: `hh outdated`, `hh update @acme/*`; каналы: pinned (по умолчанию) / auto-minor; уведомления в веб + weekly digest. Ответ на JTBD-2.
- **Approval flow**: внутренний publish → `hh gate` обязателен → approve админом → доступен команде. `quality_gates.required_checks` в манифесте уже есть — сюда и встаёт.
- **Аудит и permissions-отчёт**: журнал pull/publish/gate по людям + сводка риск-профилей всех харнесов орга («3 харнеса имеют shell-доступ, 1 — money_movement»). Риск-модель уже реализована в `hh risk` — это отчёт поверх готового.
- **Fork с provenance**: публичный харнес → внутренняя копия со ссылкой на источник и его версию; при обновлении апстрима — уведомление «ваш форк отстал, вот diff» (`hh diff` уже умеет).
- **Org-токены** для CI: пайплайн тянет приватные харнесы по scoped-токену (read-only, per-repo).
- **Import wedge (ключ к адопшену)**: «принесите ваш git-репо со скилами» — синк существующего `.claude/`-репо или plugin-marketplace в воркспейс. Команды уже живут в git; мы не просим переезжать, мы даём витрину, версии и аудит поверх их репо.

### Почему Teams продаётся раньше creator-денег

B2B-деньги надёжнее маркетплейсных (вывод из анализа рынка: авторы на файловых маркетплейсах — медиана <$50/мес; команды платят $/seat стабильно). Teams MVP = orgs + private visibility + setup-бандл + аудит — продаваемый продукт без единой строчки платёжной интеграции с авторами.

## 3. Что технически доделать (по фазам)

Инвентаризация кода (2026-07-06): auth через Supabase JWT есть (`apps/harness-api/src/server.ts:202–239`), evals/gate/risk/diff реализованы в CLI, харнесы лежат **на диске** (`seed-harnesses/`, `data/imports/`), таблиц в Supabase три (`profiles`, `user_harness_actions`, `harness_thread_posts`), социальные сигналы в API **считаются детерминированно из имени** (`computeSocial()`, server.ts:315), треды в API мокаются. Платежей, оргов, visibility, entitlements — нет нигде (grep чистый).

### Фаза 0 — фундамент честности (до денег; ~1–2 недели)

Деньги поверх мок-сигналов = trust-театр, который убьёт репутацию при первом же разоблачении.

- **P0.1 Настоящая социалка.** API читает stars/forks/runs из `user_harness_actions` и треды из `harness_thread_posts` вместо `computeSocial()`/`threadFor()`. Heat пересчитать на реальных данных, кривую холодного старта сгладить бейджами «new», а не выдуманными звёздами.
- **P0.2 Харнесы в БД.** Таблица `harnesses` (owner, name, version, visibility, manifest jsonb, archive_ref, published_at) + immutable версии архивов. Сейчас archive отдаёт текущее состояние директории — без версионных снапшотов не работают ни M2 (подписка на обновления), ни пиннинг, ни provenance форков.
- **P0.3 Манифест-расширение** (`packages/harness-schema`): `visibility: public|org|private`, `pricing: { model: free|one_time|subscription|per_call|gate_escrow, amount, currency }`, `org`. Обратная совместимость: все поля опциональны, default = public/free.

### Фаза 1 — первые деньги: M2/M2b + фиат (~2–4 недели)

- **P1.1 Таблицы**: `purchases`, `subscriptions`, `entitlements`, `payout_accounts` + RLS.
- **P1.2 Entitlement-middleware** на `GET /repos/:owner/:repo/archive` (сейчас полностью публичный): paid-харнес без entitlement → 402 с двойным payload (checkout_url + x402 requirements).
- **P1.3 Paddle/LemonSqueezy** checkout + webhook `/webhooks/paddle` → grant entitlement.
- **P1.4 CLI**: `hh pull` шлёт `HH_TOKEN` (сейчас токен используется только в `hh publish`); понятная ошибка 402 с checkout-ссылкой; `hh login` — пока copy-paste токена (как сейчас), device-flow позже.
- **P1.5 Выплаты авторам**: первые месяцы вручную (авторов мало — это фича, не баг), затем собственный сетлмент нашей фин-инфрой (фиат + USDC) — см. раздел 5; Stripe Connect не нужен.
- **P1.6 Creator storefront + attribution**: страница `@handle` как магазин креатора, checkout-ссылки/промокоды, referral-атрибуция на карточку и покупку (кто привёл трафик — тот получает ставку 95%).

### Фаза 2 — Teams MVP (параллельно Фазе 1; ~3–4 недели)

- **P2.1 Таблицы**: `organizations`, `org_members` (роли), `org_tokens` (scoped, read-only) + RLS на `harnesses.visibility=org`.
- **P2.2 API**: CRUD оргов/членов, entitlement-проверка членства на archive, журнал аудита (append-only `org_audit_log`).
- **P2.3 Setup-бандлы**: тип манифеста `bundle` (список харнесов + пины + конфиги целевых CLI), `hh setup @acme`.
- **P2.4 UI**: Network Neighborhood — окно воркспейса, замочки, permissions-сводка орга.
- **P2.5 Git-import**: синк существующего репо со скилами/plugin-marketplace в воркспейс (webhook или `hh sync`).
- **P2.6 Биллинг сидов** через Paddle (та же интеграция, что P1.3).

### Фаза 3 — agent-native оплата: x402 (~2–3 недели, поверх MCP-плана)

- **P3.1 x402 на archive**: PaymentRequirements в 402-ответе, verify/settle через CDP facilitator, entitlement на кошелёк.
- **P3.2 `hh pull --pay`**: подпись платежа локальным кошельком (`HH_WALLET_KEY`), EIP-3009 authorization.
- **P3.3 Листинг в x402 Bazaar/agentic.market** — их индекс сам подхватывает живые 402-endpoints; попадаем в каталог Coinbase раньше, чем они дорастут до харнесов.
- **P3.4 MCP-сервер** (`/mcp` из agent-first плана): search/details бесплатно, pull платного возвращает payment info — покупка внутри агентской сессии.

### Фаза 4 — дифференциаторы: gate-эскроу, endpoints, bounties

- **P4.1 Подписанные gate-receipts**: `hh gate --receipt` → JSON (harness, version, хэш результатов, timestamp) с подписью, `POST /receipts`. Сейчас `hh gate` выдаёт только текст и exit-код — receipt это новый артефакт.
- **P4.2 Эскроу-логика M3**: purchase `reserved` → pass-receipt → `captured`; fail-receipt или таймаут 72ч → `refunded`. Анти-фрод: refund только по receipt, rate-limit, спорное — вручную.
- **P4.3 Hosted endpoints (M1)**: хостинг «мозгов» как у MCPize — решение build vs partner отдельно, когда появится спрос от авторов.
- **P4.4 Bounties (M4)**: таблица `bounties` + эскроу из P4.2 + приёмка через evals.

## 4. Метрики и риски

**Метрики по фазам**: TTFI (time-to-first-install у нового пользователя), конверсия платного pull после 402, GMV и run-rate выплат авторам, org seats и weekly active pulls в воркспейсах, доля покупок через gate-эскроу и его gate-pass rate (низкий pass rate = плохие харнесы в платном сегменте — сигнал курировать).

**Риски**: (1) включить деньги до P0.1 — репутационная бомба; (2) Anthropic запустит официальную монетизацию — ответ: мульти-CLI, agent-native 402, gate-эскроу; (3) x402-спрос не материализуется — базовый сценарий выручки строим на фиате (Teams), крипто-рельсы как опцион; (4) авторов-звёзд мало — первых 10–20 приводим вручную bounty-ями и 90% фаундер-ставкой.

**Что проверить кастдевом до старта Фазы 1** (маппинг на [бриф](../research/custdev-brief-2026-07.md)): вопрос 9/40 — реальная готовность платить; вопрос 10 (короткой версии) — какие trust-сигналы нужны для платного pull; вопросы 7, 22–26 — спрос на Teams; вопрос 37 — барьеры платной публикации.

## 5. GTM: creator-led рост + собственная фин-инфра как unfair advantage

### Тезис

Продвижение не покупаем — его приносят авторы с готовой аудиторией на X/Reddit/YouTube/Telegram. Им нужен инструмент правильно продавать свои воркфлоу и системы; мы даём витрину и **весь финтех до сетлмента** (наша команда — финтех с лицензиями). Они зарабатывают → приводят своё комьюнити → часть комьюнити становится авторами. Классический flywheel двустороннего маркетплейса, где cold-start решён с обеих сторон: анкор-креатор приносит и supply (свой каталог), и demand (свою аудиторию).

### Прецедент: Whop — это ровно наша модель

- $2.67B кумулятивного GMV к февралю 2026, ~$100M GMV/мес, 183k продавцов, ~$3B годовых выплат креаторам в 144 странах; средний активный креатор зарабатывает **$8.4k/мес**, 258 продавцов заработали $1M+.
- **Урок №1 — двухуровневая комиссия**: 3% когда трафик привёл креатор своей ссылкой, 30% когда покупателя привёл маркетплейс (позже Whop опустил маркетплейс-комиссию до 0%, чтобы залить supply). Мы копируем структуру: 5% / 15%, анкорам — 0% на 6 месяцев.
- **Урок №2 — продаются outcomes, не файлы**: рабочие цены $19–39 (impulse-порог), бандлы $29 core + $79 premium. Наш gate-эскроу «плати, если evals прошли» — это outcome-продажа, доведённая до предела.
- **Урок №3 — комьюнити-гейтинг**: у Whop половина ценности — доступ в закрытые Discord/Telegram по подписке. Нам нужен entitlement-API для гейтинга каналов (см. ниже) — креаторы с комьюнити без этого не переедут.

### Что даём креатору (Creator Program)

1. **Витрина**: страница `@handle` как личный магазин (бандлы, промокоды, своя тема); Win98 share cards (`harness_flex.exe`) — мемный маркетинговый актив для постов в X, каждый шер — бесплатная реклама и нам, и ему.
2. **Фин-инфра (наш козырь)**: собственный сетлмент фиат + USDC, merchant-of-record — налоги, чарджбеки, KYC, выплаты по миру на нас. Конкуренты сидят на Stripe Connect (Agensi, MCPize) и отдают маржу платёжному стеку; мы её оставляем себе — поэтому можем держать комиссию 5% и всё равно зарабатывать. Опцион: свой **x402-facilitator** на наших лицензиях — маржа с verify/settle и «settlement-as-a-service» для чужих агентских маркетплейсов.
3. **API-first дистрибуция**: headless checkout, entitlement API + webhooks — креатор продаёт со своего лендинга/из своего бота, OnlyHarness работает как «Stripe для агентских воркфлоу» под капотом. Ключевая фича для комьюнити-креаторов: **гейтинг Discord/Telegram-доступа по entitlement** (купил харнес → бот пустил в закрытый канал).
4. **Trust-слой как анти-грифт**: evals/gate-бейдж отличает нас от потока prompt-мусора на Gumroad. Для dev-аудитории X это критично: «у меня воркфлоу с проверяемым результатом, вот gate-отчёт» — защищает репутацию самого креатора и нашу.
5. **Affiliate-петля**: члены комьюнити перепродают с процентом — вторая ступень flywheel.

### Экономика сделки с анкорами

Первые 5–10 креаторов с аудиторией: 0% комиссии на 6 месяцев (наша выручка — платёжная маржа), ручной онбординг каталога (мы сами упаковываем их воркфлоу в харнесы с evals), co-marketing (их лонч = наш лонч), персональный дашборд воронки (пост в X → карточка → pull → выручка). Питч анкору одной строкой: «средний активный креатор Whop делает $8.4k/мес на курсах и сигналах — твоя аудитория девелоперская, твой продукт проверяем evals-ами, а весь финтех-гемор мы забираем».

### Что это меняет в фазах

- Фаза 1 приоритезирует P1.6 (storefront + attribution) и переводит P1.5 на собственный сетлмент вместо Stripe Connect.
- **Ф2.5 (новая, быстрая)**: Discord/Telegram-гейтинг по entitlement API — маленькая фича, главный магнит для комьюнити-креаторов.
- Юридическая предпосылка Фазы 1: **заключение по нашим лицензиям** — покрывают ли они MoR для чужого цифрового контента, какие юрисдикции продавцов/покупателей, категория digital goods у карточных схем, политика чарджбеков. Это первый шаг, до кода.

### Риски честно

1. **MoR-ответственность**: продавая чужой контент как merchant of record, отвечаем за возвраты, фрод и сам контент. Лицензии надо проверить именно под эту активность — юрзаключение до запуска, не после.
2. **Концентрация на анкорах**: уход 2–3 креаторов = обвал GMV. Ретеншн: сетлмент-удобство, affiliate-сеть, evals-бейдж, гейтинг комьюнити — издержки переключения растут с каждой фичей.
3. **Грифтер-риск**: аудиторные AI-инфлюенсеры бывают токсичны для доверия dev-аудитории. Куратим анкоров вручную; gate обязателен для листинга в каталоге (в продажах по своей ссылке — его репутация, наш каталог — наша).
4. **Канальный конфликт с Teams** — нет: креаторы дают supply и трафик, команды — enterprise-спрос; воронки не пересекаются, а креаторские харнесы становятся контентом для Teams-каталогов.

## 6. Плагин onlyharness: агент сам подбирает, ставит и покупает

### Что это

Устанавливаемый плагин для Claude Code (позже — Codex/Cursor через AGENTS.md + `npx onlyharness`): скилл авто-подбора + MCP-конфиг на наш реестр в одном артефакте. Когда пользователь даёт агенту задачу, для которой существует специализированный харнес/воркфлоу, плагин сам находит его в реестре, показывает trust-сводку и применяет: бесплатный — сразу, платный — через покупку (checkout для человека, x402 для агента с кошельком).

### Флоу

```
Задача пользователя
  → скилл-триггер «нужен специализированный воркфлоу»
  → поиск в реестре (MCP/REST) → топ-3 кандидата
  → trust-сводка: security-verdict, Standard-бейдж, evals, permissions, context-cost
  → подтверждение пользователя
  → scoped-установка (в проект или временно на сессию) → применение
  → платный: 402 → цена + checkout_url → оплата → retry
```

### Почему это стратегически важно

1. **Реестр перестаёт быть «местом, куда ходят», и становится каналом, который сам приходит** — конверсия в момент потребности, а не в момент браузинга.
2. Каждая установка плагина = постоянный дистрибуционный канал: следующая продажа автора происходит внутри чужой агентской сессии без маркетинга.
3. Телеметрия подбора (предложено/принято/применено) — одновременно GTM-воронка и датасет качества matching-а.
4. Для creator-GTM: «поставь мой плагин» проще питчится аудитории, чем «ходи на сайт»; плагин при этом продаёт весь каталог.

### Правила безопасности (не обсуждаются)

Авто-применение чужих инструкций = поверхность prompt-injection, поэтому: плагин никогда не ставит без явного подтверждения пользователя; не предлагает харнесы без пройденного security-scan; permissions всегда показываются до установки; платное — только через explicit-покупку. Это же делает security-scan (M0) жёстким пререквизитом плагина.
